# Project Context (KT-Market Repo)

## Core Stack
- **Next.js 16.0.6+ (App Router)**
- **TypeScript (strict)**
- **Tailwind CSS v4 + CSS Variables** (`src/app/globals.css`)
- **Supabase (SSR 패키지 기반 쿠키 인증)**
- **Client state**: Zustand (`useUserStore`)
- **Server state**: TanStack React Query
- **Auth state**: Context API (`AuthContext`) + Middleware  [oai_citation:1‡GitHub](https://github.com/JunTell/KT-Market)

## Routing & Areas (src/app)
- `(home)` Route Group: 일반 사용자 영역 (로그인/회원가입, 이벤트 상세, 메인 등)
- `admin`: 관리자 대시보드 (/admin) — 이벤트/상품 관리
- `api`: 세션 처리/로그아웃 등 Auth 관련 Route Handlers  [oai_citation:2‡GitHub](https://github.com/JunTell/KT-Market)

## Auth & Security Model
- `middleware.ts`에서 Supabase 세션을 확인
- `/admin`은 “로그인 여부”뿐 아니라 DB의 `profiles.is_admin`을 조회해 **권한을 엄격 체크** 후 리다이렉트  [oai_citation:3‡GitHub](https://github.com/JunTell/KT-Market)

## Supabase Client Split (중요)
- `src/lib/supabase/`에 **Client용 / Server용 / Middleware용** 클라이언트가 분리되어 있음  
  → 환경에 맞는 클라이언트를 반드시 사용  [oai_citation:4‡GitHub](https://github.com/JunTell/KT-Market)

## Repo Structure (요약)
- `src/actions/` : Server Actions (예: `auth.ts`)
- `src/components/` : `admin/`, `auth/`, `common/`, `layout/`
- `src/lib/` : `supabase/`, `apiClient.ts`, `seo.ts`
- `src/stores/` : Zustand
- `types/` : 전역 타입 정의  [oai_citation:5‡GitHub](https://github.com/JunTell/KT-Market)

# Non-Negotiable Constraints

1. **Server vs Client Components를 항상 구분**
   - 기본은 Server Component
   - 사용자 인터랙션/React Query 훅/브라우저 API 사용 시 `"use client"` 명시

2. **Supabase 보안 원칙**
   - 서버에서 필요한 데이터는 Server Component / Server Actions에서 처리
   - 클라이언트에서 직접 쿼리하면 **RLS 정책/권한**을 더 엄격히 가정
   - 서비스 역할 키/민감한 환경변수는 절대 클라이언트로 노출 금지
   - `/admin` 권한은 “UI 숨김”이 아니라 “서버/미들웨어/DB 레벨”에서 강제

3. **TypeScript**
   - strict typing 유지
   - `any` 금지, 필요 시 `unknown` + 좁히기
   - Supabase 반환 타입/DTO 타입은 `types/` 또는 `src/types/`에 정의/재사용

4. **Styling**
   - Tailwind v4 + globals.css 변수 기반 디자인 시스템 준수 (`--primary`, `--label-900` 등)  [oai_citation:6‡GitHub](https://github.com/JunTell/KT-Market)

# Answering Playbook (반드시 이 순서로)

## 1) Analyze (설계 판단)
- “왜 Server/Client를 나누는지”
- “어떤 폴더/레이어에 넣는지” (예: `src/actions`, `src/lib`, `src/components`)
- “React Query vs Server fetch vs Zustand 중 무엇을 쓰는지”를 근거와 함께 설명

## 2) Safety Check (보안 점검)
- 세션/쿠키 처리, RLS, 관리자 권한 체크 누락 여부
- 입력값 검증(특히 Server Actions/Route Handlers)
- 공개되면 안 되는 데이터/환경변수/토큰 노출 여부

## 3) Code (모듈형/리뷰 가능한 코드)
- **항상 파일 경로를 주석으로 명시**
  - 예: `// src/app/(home)/page.tsx`
- 기존 구조를 우선 사용
  - 데이터 변이: `src/actions/*`
  - Supabase 클라이언트: `src/lib/supabase/*` (환경별 올바른 클라이언트)
  - 공통 유틸: `src/lib/apiClient.ts`, `src/lib/seo.ts` 재사용
- 필요 시 사용
  - 아이콘: `lucide-react`
  - 검증: `zod`

## 4) Review (학습 포인트 정리)
- Next.js App Router 기능(예: Server Actions, Route Handlers, `revalidatePath`, `redirect`, `cookies`)을 썼다면
  - “왜 썼는지 / 대안은 무엇인지 / 흔한 실수”를 짧게 설명
